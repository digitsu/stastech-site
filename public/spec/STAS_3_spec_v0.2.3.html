STAS 3.0 — Token Protocol Specification (Draft)

Version: 3.0. (freeze, confiscation, swap, multisig, swap transaction size optimization)

1. Scope

This specification defines the on-chain of the STAS 3.0 tokenised UTXO protocol, implemented entirely within Bitcoin SV (BSV) Layer-1 Script. It covers:

  • The layout of a STAS 3.0 locking script (token frame).

  • The encoding and semantics of every variable-length field.

  • The parameter ordering of the corresponding unlocking script.

  • The permitted state transitions (spend, split, merge, swap, freeze, confiscate, swap-cancel).

  • Numeric opcode-level protocol constants.

It does not specify issuance policy, indexer behavior, fee economics, or off-chain wallet conventions.

3. Terminology

Term Meaning
Frame A single STAS 3.0 UTXO (locking script + satoshi value).
Engine The fixed 2.9 Kbytes opcode body that implements the STAS state machine; identical across all tokens of an issuance.
Variable field A data push whose content is permitted to change across spends.
MPKH Multi-Public-Key Hash — HASH160 of a P2MPKH redemption script. May substitute for a public key hash of P2PKH anywhere the protocol accepts an address.
Sighash preimage Concatenation defined by BSV sighash rules; consumed by the engine for self-verification via the "ECDSA trick".
Service field Auxiliary data block whose presence is conditional on a flag bit.
Protocol ID The 20-byte redemption-address slot; doubles as a cross-UTXO identifier of a token issuance.

4. Protocol Overview

A STAS 3.0 token is one UTXO whose locking script has the form:

<owner> <var2> [FIXED ENGINE OPCODES] OP_RETURN <protoID> <flags> <svc…> <data…>

Only the two leading pushes (owner, var2) are allowed to change across spends. Any valid state transition produces one or more new outputs whose fixed-engine region is byte-identical to the input's engine region; the engine itself enforces that invariant via “sighash preimage” inspection.

High-level operations:

Op Spending-type Transaction-type
Regular spend / split 1 0
Atomic swap 1 1
Merge 1 2–7
Freeze / unfreeze 2 0
Confiscate 3 anything
Swap cancellation 4 0

5. Locking Script (Frame) Format

Fields are listed in byte order. All pushes use minimal Bitcoin Script push encoding unless otherwise stated.

5.1 Executable Region

# Field Length Encoding Content
1 owner 20 B 0x14 <bytes>

1.HASH160(pubkey) (PKH)

2. HASH160(MPKH redemption script). 3. The special value HASH160("") (= b472a266d0bd89c13706a4132ccfb16f7c3b9fcb) - disables signature verification for this UTXO.
2 var2 maximum pushable value

  1. Single push op. Permitted forms: OP_0, OP_1…OP_16, OP_1NEGATE,

  2. direct push (len ≤75),

  3. OP_PUSHDATA1/2/4.

Must obey minimal-encoding.

 Second variable field — see §6.
3 engine ~2.9 kB Fixed opcode sequence (see template). STAS 3.0 state-machine code. Byte-for-byte invariant across spends.
4 OP_RETURN 1 B 0x6a Marks end of executable script.

5.2 Data Region (after OP_RETURN)

The data region is never executed, but is inspected by the engine of the spending transaction via the sighash preimage. Fields are concatenated in order:

5.2.1 Protocol ID / Redemption Address
Field Length Content
protoID 20 B HASH160(pubkey) or HASH160(MPKH redeem script) designating the issuer / redemption target. Acts as a stable immutable issuance fingerprint across all UTXOs of the issuance.
5.2.2 Flags Field
Field Length Encoding Content
flags 1 B … 255 B (≤75 B recommended) Single “push” value Bit-field expressing optional capabilities.

Flag bit assignment (rightmost bit = bit 0):

Bit Name Meaning
0 FREEZABLE This issuance permits freeze/unfreeze directly by the freeze authority.
1 CONFISCATABLE This issuance permits confiscation directly by the confiscation authority.
2–N reserved reserved

Permitted value encodings (examples):

  • FREEZABLE only → push 0x01, or push 0x0001 etc.

  • CONFISCATABLE only → push 0x02, or push 0x0002 etc.

  • Both → push 0x03, or push 0x0003 etc.

  • None → 0x00; or empty push: OP_0/OP_FALSE; or absent in case no data attachment follows it.

5.2.3 Service Fields

Each field is present only when its corresponding flag bit is set. Serialized in left-to-right order of increasing flag bit — i.e. the opposite of bits order within the flags field (which is right to left, starting from bit 0).

Condition Field Length Content
FREEZABLE set freezeAuth 20 Bytes HASH160 of freeze authority (PKH or MPKH).
CONFISCATABLE set confiscateAuth 20 Bytes HASH160 of confiscation authority (PKH or MPKH).

Authorities are fixed at issuance and immutable thereafter.

5.2.4 Optional Data
Field Length Content
payload MAX locking script size/MAX transaction size Arbitrary issuer data (metadata, legal pointers …). Not interpreted by the engine (rather on the higher level).

6. Second Variable Field (var2) Sub-formats

The first byte of var2 (after its push header) is an action identifier selecting one of three sub-formats.

6.1 Passive / Data-only (action = 0x00 or empty push)

Byte range Field Meaning
0 0x00 (or empty push) Signals no active state.
1…n arbitrary Assigned by UTXO owner at his discretion. Treated by the script as tag-along only; may be used for per-UTXO notes or higher level code.

6.2 Frozen Marker (action = 0x02)

Produced by the freeze operation (§9.2). Construction rules:

Original var2 form Frozen var2 form
empty push (OP_0) OP_2 (opcode 0x52)
pushdata bytelength, OP_PUSHDATA1/2/4 Prepend 0x02 to the pushed bytes
OP_1, OP_3…OP_16, OP_1NEGATE Convert to pushdata form (values pushed on stack by these opcodes), then prepend 0x02

Unfreezing (§9.2) strictly reverses the above mapping.

6.3 Swap Descriptor (action = 0x01)

Fixed 61-byte minimum layout:

Offset Field Length Encoding Content
0 action 1 B 0x01 Swap marker.
1 requestedScriptHash 32 B raw SHA256 (single) of the full locking script the maker is willing to accept aka requested asset.
33 receiveAddr 20 B raw HASH160 of the address/MPKH to which the counter-asset is delivered. Also identifies the maker for cancellation (§9.4).
53 rateNumerator 4 B little-endian uint32 Numerator of exchange rate.
If numerator is zero, script skips rate check
57 rateDenominator 4 B little-endian uint32 Denominator of exchange rate.
61 next (var2 value after the swap) variable optional Any next var2 value after the swap or may be absent at all.
E.g. recursive swap descriptor (see below).

Exchange rate semantics: given an input amount A of the wanted asset, the output amount delivered to receiveAddr is:

A' = A × (rateNumerator / rateDenominator)

The length of the amount/value field in Bitcoin (both A' and A in the formula) is 8 bytes.
It is multiplied first to scale the dividend, in order to preserve more information before division, which reduces remainder loss.

Note, next field is simply contains the value swap maker requires to be assigned (by swap taker) after the swap into 2nd variable field of his new UTXO (in which he receives his requested assets).

As a specific use-case, next permits a nested list of configurations for recursive swaps through a chain of (action byte, requestedScriptHash, receiveAddr, rate, next) nested tuples, allowing a received swapped assets to be right away configured to be swapped again with next configuration at will.
Encoding is the same as the top-level descriptor, minus including the leading action byte. Note, the terminal element carries only (action byte, requestedScriptHash, receiveAddr, rate) with no further next field.

Parameters are pushed in the order listed. "Empty" denotes OP_FALSE/OP_0 empty string push.

15 noteData 0 … 65 533 B or OP_FALSE Payload for trailing OP_FALSE OP_RETURN note output.
16 fundIdx 4 B LE or OP_FALSE Output index (in funding transaction) spent through this input (prev-vOut).
17 fundTxid 32 B or OP_FALSE TXID of funding transaction, which output is spent through this input (prev-tx).

21… authz variable

  1. P2PKH:
    Signature + public key

  2. P2MPKH:
    <multisig-dummy OP_0> + <signature/s array> + <P2MPKH redemption script>.

  3. To skip auth:
    single emtpy string push (by OP_0/OP_FALSE).

In swap and merge transactions, additional parameters follow the above block (see §9.3, §9.5).

1st STAS output is a mandatory one, for which 3 first parameters values are always passed/present. While following 2-4 outputs are optional STAS split-outputs, the absence of these 2–4 outputs is not designated by passing any specific values as e.g. OP_0/OP_FALSE instead of their triplet parameters — all their related triplets are simply skipped in unlocking script, treated as non-existing.

8. Control Codes

8.1 Transaction-type (txType)

Value Meaning Additional trailing params
0 Regular spend (split or 1-in/1-out) none
1 Atomic swap

Counterparty’s:

  1. locking script,

  2. transaction pieces count,

  3. transaction pieces array
2 Merge, 2 pieces piece count (2), array of 2 pieces
3 Merge, 3 pieces piece count (3), array of 3 pieces
4 Merge, 4 pieces piece count (4), array of 4 pieces
5 Merge, 5 pieces piece count (5), array of 5 pieces
6 Merge, 6 pieces piece count (6), array of 6 pieces
7 Merge, 7 pieces piece count (7), array of 7 pieces

The piece count is passed explicitly and its value in the range [2-7] designates both merge transaction type and the number of transaction pieces in array (of the pieces) following it (to be used in counterparty’s transaction reconstruction).
In the array of pieces of the preceding transaction each piece is its own OP_PUSHDATA operation in the unlocking script (in ASM text these are space-separated, for each piece to be a separate push) and the order of the pieces appearance is reversed.

8.2 Spending-type (spendType)

Value Meaning
0 Reserved / undefined.
1 Regular spend, split, merge, or swap.
2 Freeze or unfreeze (requires freeze authority).
3 Confiscation (requires confiscation authority).
4 Swap cancellation (requires swap initiator/maker authorization).

9. Operational Semantics

9.1 Regular Spend / Split

  • txType = 0, spendType = 1.

  • Up to 4 STAS outputs may be produced.

  • Each STAS output copies the engine region byte-for-byte and sets its own owner and var2 (unless specific conditions enforce their values, e.g. divisible swap).

  • Sum of STAS output amounts MUST equal sum of STAS input amounts.

  • Signature validates against the input UTXO's owner under P2PKH or P2MPKH unlocking format.

9.2 Freeze / Unfreeze

  • spendType = 2.

  • Exactly one STAS output.

  • owner and all other data fields MUST be byte-identical to the input, except var2, which MUST be rewritten/prepended per §6.2 (forward for freeze, reverse for unfreeze).

  • Signature/authorization MUST validate under the freezeAuth service field, which must be present.

  • The FREEZABLE flag MUST be set in the input frame.

9.3 Confiscation

  • spendType = 3.

  • No restriction on number of outputs or on owner and var2 fields of outputs.

  • Confiscation authority may concurrently freeze the destination frame.

  • Authorization MUST validate under confiscateAuth service field, which must be present.

  • CONFISCATABLE flag MUST be set in the input frame.

  • Highest precedence of all operations.

9.4 Swap Cancellation

  • spendType = 4.

  • Single output whose owner equals the input's var2.receiveAddr.

  • Authorization MUST validate under that same receiveAddr (P2PKH or P2MPKH unlocking format).

  • Input var2 MUST be a swap descriptor (action 0x01).

  • Cannot be combined with an executing swap in the same transaction.

9.5 Atomic Swap

  • txType = 1, spendType = 1 on both STAS inputs.

  • Both inputs' var2 may carry compatible satisfied (by constructed transaction) swap descriptors; or at least one of them MUST carry swap descriptor with configuration, which is satisfied by the counterparty lacking such a swap descriptor.

  • Counterparty script is supplied as a trailing unlocking parameter together followed by a piece count and (followed by) the reverse-ordered array of pieces that remain after excising the counterparty asset script form its preceding transaction.
    The excised script constitutes all the token data right after the two variable parameters at the very start of the STAS script.
    In the reverse-ordered array of pieces of the preceding transaction each piece is its own OP_PUSHDATA operation in the unlocking script (in ASM text these are space-separated, for each piece to be a separate push).

  • Output index assignment:

    • Requested asset is received at the output matching the initiator's input index.

    • Given asset is delivered at the output matching the opposite (counterparty) input index.

    • Remainder / split outputs inherit the source UTXO's both owner and var2 fields.

  • If either owner field equals HASH160("") (=b472a266d0bd89c13706a4132ccfb16f7c3b9fcb), no signature from that side (leg) is required (arbitrator-free swap), i.e. that input is spent without any signing involved.

  • An arbitrator is an optional entity/entities designated by a swap party (one of both) via assignment of arbitrator’ address/MPKH in owner field to be required to sign their leg of the swap.

  • When both parties have swap configurations in their UTXOs, and both of them assigned specific arbitrator/s requirements, each of the legs of the swap enforces presence of signature/s of its corresponding required arbitrator/s (assigned in their 1st variable fields). 

9.6 Precedence

From highest to lowest, when multiple modes appear applicable:

  1. Confiscation

  2. Freeze / unfreeze

  3. Swap

  4. Regular spend

Freeze and confiscate MUST NOT burn tokens.
Only the issuer/s, after receiving the tokens (from token owner/s) to protoID, may redeem supply by spending it from protoID (to any other address/MPKH at his/their discretion).

10. Authorization Schemes

10.1 P2PKH (classical)

  • owner = HASH160(pubkey), 20 B.

  • Unlocking: <sig> <pubkey>.

  • Engine validates OP_HASH160 (pubkey) == owner and OP_CHECKSIG.

10.2 P2MPKH (Pay-to-Multi-Public-Key-Hash)

  • owner = single 20-byte MPKH -> HASH160(P2MPKH redeem script), variable length.

  • Unlocking: dummy value OP_0, array of <sig>, P2MPKH redeem script

  • Engine validates fixed 70-byte P2MPKH locking script:
    OP_HASH160 (P2MPKH redeem script) == owner and OP_CHECKMULTISIG

The fixed 70-byte P2MPKH locking script (for n <=5):

OP_DUP OP_HASH160 <MPKH> OP_SIZE

21 OP_EQUAL OP_IF

OP_CHECKSIG

OP_ELSE

OP_1 OP_SPLIT

OP_1 OP_SPLIT OP_IFDUP OP_IF OP_SWAP OP_SPLIT OP_ENDIF

OP_1 OP_SPLIT OP_IFDUP OP_IF OP_SWAP OP_SPLIT OP_ENDIF

OP_1 OP_SPLIT OP_IFDUP OP_IF OP_SWAP OP_SPLIT OP_ENDIF

OP_1 OP_SPLIT OP_IFDUP OP_IF OP_SWAP OP_SPLIT OP_ENDIF

OP_1 OP_SPLIT OP_IFDUP OP_IF OP_SWAP OP_SPLIT OP_ENDIF

OP_CHECKMULTISIG

OP_ENDIF

Important note: fixed P2MPKH locking script format as provided here, is used explicitly only before and after STAS token’s lifecycle, namely at issuance and redemption UTXOs (preceding and terminating its life-cycle).
STAS token outputs implement exactly same logic of this fixed P2MPKH script template, but do not have it as explicit sub-template in them.

  • Hex fragments embedded in the canonical engine: 4676a914 <MPKH> 888201218763ac67517f517f73637c7f68517f73637c7f68517f73637c7f68517f73637c7f68517f73637c7f68ae68
    (prefix 0x46 = 70-byte script length).

  • n ≤ 5 public keys (5 is max in the current version); expandable at ~7 byte per additional key.

  • m = required signatures (out of n); threshold.

  • Unlocking stack for multisig path: OP_0 <sig_1> … <sig_m> <redemption_script>.

  • <redemption_script>, similarly to multisig locking script constitutes of:
    m <pubkey_1> … <pubkey_m> n
    note, public keys are prepended by length prefix.
    An example:
    ‘03’ + ‘21’ + publicKey1 + ‘21’ + publicKey2 + ‘21’ + publicKey3 + ‘21’ + publicKey4 + ‘21’ + publicKey5 + ‘05’
    // where 0x21 stands for pushdata of compressed public key length of 33 bytes (preceding public key itself)
    // first byte: m (required signatures) — ‘03’
    // last byte: n (total public keys) — ‘05’
    // public keys only are prepended by length prefix (as in ASM format).

  • Single-sig spend path is taken when the supplied "pubkey" is exactly 33 bytes (size 0x21), making individual spends from an MPKH address indistinguishable on-chain from a normal P2PKH spend.

10.3 Signature-Suppression Mode

Setting owner to HASH160("") = b472a266d0bd89c13706a4132ccfb16f7c3b9fcb instructs the engine to accept OP_FALSE in place of all auth unlocking script parameters – dummy-value (if it’s P2MPKH spending case) signature/s and “public key”/”P2MPKH redemption script”, in order to skip all ECDSA checks for that party. Used in divisible swaps to indicate swap as arbitrator-free. Setting (at swap configuration) var2.receiveAddr to HASH160("") may be used to indicate the same arbitrator-free swap type for a next iteration in nested recursive swap configuration case.

11. Trailing Per-transaction Note Output (Data Memo)

Every STAS 3.0 transaction MAY carry exactly one trailing "note" output of the form:

OP_FALSE OP_RETURN <payload>

Field Limit
payload ≤ 65 533 B (0xFFFF − 2, reserving two opcodes)

If no note is desired the unlocking script MUST supply a single OP_FALSE in the noteData slot.

13. Migration Notes (from STAS 2.0)

Informative.

  • Issuance / redemption outputs become P2MPKH rather than P2PKH locking scripts, requiring wallets to update spending format accordingly.

  • Script identification logic MUST account for the variable length of var2; a fixed-offset scan will no longer locate the engine region.

  • Both assets in a swap may now split; STAS 2.0 permitted splitting on only one side.

  • Unlocking scripts gain:

  1. a third parameter per STAS output (var2) (couples turned into triplets per STAS output),

  2. a new spendType parameter,

  3. a restructured transaction-type swap case (value ‘1’) trailing parameters (instead of whole preceding transaction -> counterparty script, piece count, array of pieces).

14. Notes

If protoID is changed, the ID of the token changes, to issue various assets under the same main issuance ID, sub-field should be added by the issuer inside data-attachment (behind OP_RETURN) for higher-level software to interpret its sub-type.
